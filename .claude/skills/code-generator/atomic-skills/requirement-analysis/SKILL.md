# Requirement Analysis 原子Skill v1.2

## 概述

解析用户提供的需求文档、功能模块描述或自然语言描述，提取关键信息，识别需求类型（单模块/多模块），为后续代码生成提供准确的输入。

**v1.2 改进**：明确依赖 multi-scenario-adapter 和 requirement-reader，仅在自由文本输入场景下执行。

## 核心能力

- 解析自然语言需求
- 识别功能点和边界条件
- 区分单模块和多模块需求
- 提取输入输出规范
- 识别异常处理需求

## 输入

- **multi-scenario-adapter 输出**：包含场景类型（场景2/场景3）
- **requirement-reader 输出**：包含原始需求文本和规范化数据
- **用户对代码需求的描述**（文本）

## 输出

结构化的需求分析报告：

```markdown
# 需求分析报告 v1.2

## 基础信息
- **原始需求**：[用户原始输入]
- **需求类型**：[单模块/多模块]
- **场景类型**：[场景2-简单需求/场景3-复杂需求]
- **预计文件数**：[1个/多个]
- **分析时间**：2026-04-08 20:00:00

## 功能点分析

### 功能点1：[功能名称]
- **描述**：[详细描述]
- **输入**：[输入参数/条件]
- **输出**：[返回值/行为]
- **边界条件**：[异常情况、特殊输入]
- **依赖**：[依赖的其他功能]

### 功能点2：[功能名称]
- **描述**：[详细描述]
- **输入**：[输入参数/条件]
- **输出**：[返回值/行为]
- **边界条件**：[异常情况、特殊输入]
- **依赖**：[依赖的其他功能]

## 模块识别
- **模块划分**：[列出所有模块]
- **模块间关系**：[描述模块间的依赖关系]
- **是否为多模块**：[是/否]
- **多模块特征**：[如：多个独立功能、模块间有数据交互]

## 约束条件
- **技术栈要求**：[用户指定的技术栈]
- **性能要求**：[响应时间、并发量等]
- **安全要求**：[输入验证、权限控制等]

## 需求澄清点
- [需要用户确认的问题1]
- [需要用户确认的问题2]

## 结论
[总结需求类型和核心功能]
```

## 执行逻辑

### 步骤1：读取输入

1. 接收 multi-scenario-adapter 的场景类型
2. 接收 requirement-reader 的原始需求文本
3. 判断执行深度（简单需求 vs 复杂需求）

### 步骤2：解析需求（根据场景）

**场景2（简单需求）**：
1. 快速识别单一功能点
2. 提取输入、输出、边界条件
3. 标记为"单模块"
4. 跳过模块关系分析

**场景3（复杂/多模块需求）**：
1. 解析自然语言，提取核心功能点
2. 识别每个功能点的输入、输出、边界条件
3. 判断是否为多模块需求
4. 识别模块间依赖关系
5. 整理模块划分

### 步骤3：输出报告

1. 生成结构化需求分析报告
2. 提出1-2个需要澄清的问题
3. 输出完整的分析结果

## 依赖关系

- **依赖**：multi-scenario-adapter（场景类型）、requirement-reader（原始需求）
- **被依赖**：tech-stack-detection、code-design

## 完成标准

1. ✅ 至少识别出1个功能点
2. ✅ 明确输入、输出和边界条件
3. ✅ 正确判断为单模块或多模块需求
4. ✅ 识别模块间关系（如果是多模块）
5. ✅ 输出完整结构化报告（markdown）
6. ✅ 提出1-2个需要澄清的问题（可选）

## 错误处理

- **需求模糊**：提出澄清问题，暂停后续步骤
- **需求冲突**：指出矛盾点，等待用户确认
- **需求过宽**：建议拆分，等待用户确认
- **需求无意义**：提示用户重新描述

## 示例

### 场景2（简单需求）输入

```
用户输入："写一个函数，将 ISO 时间字符串格式化为 YYYY-MM-DD HH:mm 格式"
multi-scenario-adapter：场景2（简单需求）
```

### 场景2 输出

```markdown
# 需求分析报告

## 基础信息
- **原始需求**：写一个函数，将 ISO 时间字符串格式化为 YYYY-MM-DD HH:mm 格式
- **需求类型**：单模块
- **场景类型**：场景2-简单需求
- **预计文件数**：1个
- **分析时间**：2026-04-08 20:00:00

## 功能点分析

### 功能点1：日期格式化
- **描述**：将 ISO 格式的时间字符串转换为易读格式
- **输入**：ISO 格式字符串（如 "2026-04-08T10:30:00Z"）
- **输出**：格式化字符串（如 "2026-04-08 10:30"）
- **边界条件**：
  - 空字符串返回空字符串
  - 无效格式返回原始字符串
- **依赖**：无

## 约束条件
- **技术栈要求**：遵循项目现有代码风格
- **性能要求**：无
- **安全要求**：输入验证

## 结论
这是一个简单的单函数需求，无需模块划分，直接实现即可。
```

### 场景3（复杂需求）输入

```
用户输入："实现一个会议管理系统，包含三个模块：
1. 会议创建模块：创建会议、设置标题、开始时间、结束时间、是否需要密码
2. 会议列表模块：展示所有会议，支持按状态筛选
3. 会议详情模块：显示会议详情、编辑、删除、加入会议"
multi-scenario-adapter：场景3（复杂需求/多模块）
```

### 场景3 输出

```markdown
# 需求分析报告

## 基础信息
- **原始需求**：实现会议管理系统（创建、列表、详情）
- **需求类型**：多模块
- **场景类型**：场景3-复杂需求
- **预计文件数**：6个
- **分析时间**：2026-04-08 20:00:00

## 功能点分析

### 功能点1：会议创建
- **描述**：创建新会议，设置会议基本信息
- **输入**：标题、开始时间、结束时间、是否需要密码
- **输出**：创建成功返回会议ID
- **边界条件**：开始时间不能早于当前时间
- **依赖**：无

### 功能点2：会议列表
- **描述**：展示所有会议，支持按状态筛选
- **输入**：筛选条件（进行中/已结束/所有）
- **输出**：会议列表
- **边界条件**：无会议时显示空状态
- **依赖**：无

### 功能点3：会议详情
- **描述**：显示会议详情，提供编辑、删除、加入功能
- **输入**：会议ID
- **输出**：会议完整信息、操作按钮
- **边界条件**：会议不存在时显示404
- **依赖**：会议创建、会议列表

## 模块识别
- **模块划分**：会议创建、会议列表、会议详情
- **模块间关系**：
  - 会议创建 → 会议列表（创建后刷新列表）
  - 会议列表 → 会议详情（点击进入详情）
- **是否为多模块**：是
- **多模块特征**：三个独立功能模块，模块间有数据交互

## 结论
这是一个多模块需求，需要生成多个文件并进行模块整合。
```

## v1.2 改进点

1. **明确场景类型**：根据 multi-scenario-adapter 的判断决定分析深度
2. **简化简单需求处理**：场景2跳过模块关系分析
3. **完整复杂需求分析**：场景3详细分析所有功能点和模块关系
4. **标记来源**：报告中标注场景类型

## 注意事项

- 保持客观，不猜测未证实的细节
- 优先提取事实，再进行判断
- 多模块需求的识别是关键，直接影响后续流程
- 提出的澄清问题必须具体可回答

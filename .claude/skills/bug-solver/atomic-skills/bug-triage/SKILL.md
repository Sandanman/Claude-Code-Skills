# Bug Triage 原子Skill

## 概述
对用户报告的bug进行分类，确定bug类型和优先级，并制定处理策略。是整个bug解决流程的第一步，决定了后续的处理方式。

## 核心能力
- 分析错误信息，提取关键特征
- 判断bug类型（frontend/backend/config/logic）
- 评估优先级（critical/high/medium/low）
- 制定处理策略（立即修复/计划修复/记录跟踪）
- 决定信息收集重点

## Bug分类维度

### 按类型分类
| 类型 | 关键字/特征 | 典型错误示例 |
|------|------------|-------------|
| frontend | 页面、控制台、点击无反应、样式异常、渲染错误 | TypeError, ReferenceError, XSS, 样式错位 |
| backend | API、服务器、数据库、网络请求 | 500错误, 502错误, 连接超时 |
| config | 环境变量、配置文件、路径错误 | env变量缺失, import路径错误 |
| logic | 业务逻辑、数据处理、计算错误 | 计算结果错误, 流程不符合预期 |

### 按优先级分类
| 优先级 | 判断标准 | 响应时间 | 示例 |
|--------|----------|----------|------|
| critical | 系统崩溃、数据丢失、安全漏洞、核心功能完全不可用 | 立即处理 | 登录页面崩溃、支付失败、XSS注入风险 |
| high | 核心功能无法使用，无替代方案 | 24小时内 | 会议无法创建、用户无法注册、API返回500 |
| medium | 功能部分受损，有替代方案 | 72小时内 | 通知有延迟、列表加载慢、样式轻微偏差 |
| low | 体验问题，不影响核心功能 | 1周内 | 按钮样式偏差、轻微延迟、提示信息不准确 |

### 优先级判定决策树
```
bug发生?
├── 系统崩溃/数据丢失? → critical
├── 安全漏洞? → critical
├── 核心功能完全不可用? → high
├── 功能部分受损但有替代? → medium
└── 仅体验问题? → low
```

### 处理策略
| 策略 | 适用优先级 | 说明 |
|------|------------|------|
| 立即修复 | critical, high | 快速定位根因，优先修复 |
| 计划修复 | medium | 按标准流程进入开发计划 |
| 记录跟踪 | low | 记录到issue tracking系统 |

## 输入
用户对bug的描述（文本）

## 输出
Bug分类报告：

```markdown
# Bug 分类报告

## 分类结果

### Bug类型：frontend
- **判断依据**：
  - 错误发生在浏览器环境（控制台显示）
  - 与页面交互相关（点击按钮无反应）
  - Vue组件相关错误（TypeError涉及undefined属性访问）

### 优先级：high
- **判断依据**：
  - 核心功能不可用（登录是系统入口功能）
  - 用户无法完成基本操作
  - 无替代方案（必须登录才能使用系统）
  - 影响范围：所有用户

### 处理策略：立即修复
- **策略说明**：由于优先级为high，需要快速定位根因并修复
- **预计处理时间**：1-2小时

### 信息收集重点
由于类型是frontend，优先级是high，后续bug-identification阶段需要重点关注：
1. 完整的控制台错误信息（包括错误堆栈）
2. 浏览器版本和设备信息
3. 复现步骤（尽可能详细）
4. 相关代码文件位置

## 分类依据
- **用户原始描述**：登录页面点击登录按钮没有反应，控制台显示TypeError: Cannot read property 'login' of undefined
- **提取的关键词**：TypeError, undefined, login, 登录页面, 按钮点击无反应
- **错误类型特征**：前端错误，涉及Vue组件属性访问
- **影响范围**：所有尝试登录的用户
```

## 执行逻辑
1. 接收用户对bug的描述
2. 提取错误信息中的关键特征
3. 对比类型分类标准，确定bug类型
4. 对比优先级判定标准，确定优先级
5. 根据优先级制定处理策略
6. 明确后续信息收集重点
7. 输出完整的Bug分类报告

## 依赖关系
- 依赖：无（首个原子skill）
- 被依赖：bug-identification

## 完成标准
1. Bug类型已明确分类（frontend/backend/config/logic之一）
2. 优先级已明确判定（critical/high/medium/low之一）
3. 处理策略已制定
4. 后续信息收集重点已明确
5. 输出完整的Bug分类报告

## 错误处理
- 如果错误信息不足以判断类型，标记为"需要更多信息"，询问用户补充
- 如果错误跨越多个类型，选择影响最大的类型作为主类型

## 注意事项
- 分类要基于客观的证据，不主观猜测
- 优先级判定要综合考虑影响范围、用户数量、业务重要性
- 处理策略要与优先级匹配
- 分类结果将影响后续所有步骤的处理方式

## 示例

**输入**：
"登录页面点击登录按钮没有反应，控制台显示TypeError: Cannot read property 'login' of undefined"

**输出**：（如上所示的Bug分类报告）

**输入2**：
"用户在会议列表页面，刚创建的会议显示在列表最下面，需要手动刷新才能看到"

**输出2**：
```markdown
# Bug 分类报告

## 分类结果

### Bug类型：frontend
- **判断依据**：前端渲染/数据刷新问题，WebSocket或响应式数据更新相关

### 优先级：medium
- **判断依据**：功能部分受损（有替代方案：手动刷新），不影响创建会议本身
- **影响范围**：创建会议的用户

### 处理策略：计划修复
- **策略说明**：优先级为medium，进入开发计划按标准流程处理
- **预计处理时间**：半天

### 信息收集重点
1. 是否使用WebSocket？数据更新的推送机制是什么？
2. Vue组件的数据响应式配置
3. 相关的API响应数据结构
```

# Fix Verification 原子Skill

## 概述
验证修复方案是否真正解决了问题，并检查是否引入了新的问题。

## 核心能力
- 验证修复是否解决问题
- 检查是否引入新问题
- 对比修复前后差异
- 输出验证报告

## 输入
fix-generation输出的修复方案文档

## 输出
修复验证报告：

```markdown
# 修复验证报告

## 验证目标
验证修复是否解决了登录功能中的TypeError错误

## 验证方法

### 方法1：功能验证（主要）
- **步骤**：重启服务器 → 打开登录页 → 输入凭据 → 点击登录
- **预期结果**：登录成功跳转首页，无错误
- **实际结果**：登录成功，无错误
- **结论**：通过

### 方法2：代码审查（辅助）
- **步骤**：检查导入路径是否正确
- **预期结果**：导入路径为'@/api/user.js'
- **实际结果**：正确
- **结论**：通过

## 验证结果
| 验证点 | 结果 | 说明 |
|--------|------|------|
| 问题是否解决 | 是 | 登录成功，无TypeError |
| 是否引入新问题 | 否 | 其他功能正常 |
| 功能完整性 | 是 | 登录功能完整 |

## 验证结论
**修复成功**：问题已解决，未引入新问题。
```

## 执行逻辑
1. 根据修复方案中的验证建议进行验证
2. 执行至少两个验证方法（主要+辅助）
3. 记录修复前后对比
4. 评估是否引入新问题
5. 输出完整的验证报告

## 依赖关系
- 依赖：fix-generation（必须有完整的修复方案文档）
- 被依赖：test-suggestion

## 完成标准
1. 至少执行两个验证方法
2. 包含修复前后对比
3. 明确结论：修复成功/失败/部分成功
4. 评估是否引入新问题
5. 输出完整的验证报告

## 错误处理
- 如果验证失败，记录具体失败原因
- 如果验证环境不一致，标记为"需要在特定环境验证"
- 如果无法复现问题，标记为"无法复现"

## 注意事项
- 验证必须基于真实环境，不能仅凭代码分析
- 优先使用功能验证，代码审查作为辅助
- 所有验证结果必须可复现
- 保持客观，如实记录结果

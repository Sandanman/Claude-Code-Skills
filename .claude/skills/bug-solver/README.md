# Bug Solver Skill 使用指南

## 快速开始

当您遇到应用程序bug时，只需描述问题，bug-solver会自动帮您系统化地诊断和修复。

### 使用方式

```
用户输入bug描述 → orchestrator识别意图 → bug-solver接管任务 → 自动执行完整流程
```

### 示例输入

**示例1：前端逻辑错误**
```
登录页面点击登录按钮没有反应，控制台显示TypeError: Cannot read property 'login' of undefined
```

**示例2：后端接口问题**
```
获取会议列表接口返回500错误，错误信息：数据库连接超时
```

**示例3：配置问题**
```
Vue项目打包后访问路由出现404，刷新页面无法加载
```

## 执行流程详解

### 1. 问题识别（bug-identification）
- 收集错误日志、控制台输出
- 整理复现步骤
- 明确期望行为 vs 实际行为
- 收集环境信息（浏览器、Node版本等）

### 2. 代码分析（code-analysis）
- 分析相关代码文件
- 检查错误处理逻辑
- 识别边界条件
- 标记可疑代码段

### 3. 根因定位（root-cause-analysis）
- 基于代码分析和错误信息
- 定位具体的代码行
- 分析逻辑错误原因
- 输出根因文档

### 4. 修复生成（fix-generation）
- 根据根因设计修复方案
- 生成具体代码修改
- 提供修复说明
- 考虑向后兼容性

### 5. 修复验证（fix-verification）
- 验证修复是否解决问题
- 检查是否引入新问题
- 对比修复前后差异
- 输出验证报告

### 6. 测试建议（test-suggestion）
- 建议单元测试用例
- 建议集成测试场景
- 覆盖正常路径和异常场景
- 防止问题再次发生

## 特殊情况处理

### 问题：修复验证失败
如果fix-verification发现修复不完整，会自动返回fix-generation阶段重新生成修复方案（最多循环2次）

### 问题：原子skill执行失败
任一原子skill重试3次仍失败，会暂停任务并通知用户确认是否继续

### 问题：需要更多信息
在bug-identification阶段，如果信息不足，会提示用户补充：
- 完整的错误堆栈
- 复现步骤
- 环境配置
- 相关代码文件

## 最佳实践

1. **提供完整信息**：包含错误日志、复现步骤、期望行为
2. **明确问题范围**：指出是哪个页面、哪个功能出现问题
3. **附上环境信息**：浏览器版本、Node版本、依赖版本等
4. **确认修复方案**：在应用修复前，仔细阅读修复说明

## 输出产物

任务完成后会输出：
1. Bug分析报告（包含根因定位）
2. 修复代码补丁
3. 验证报告
4. 测试用例建议
5. 完整执行日志

所有产物会归档到 `.claude/skills/tasks/history/YYYY-MM/` 下

## 常见问题

**Q: 这个skill能解决所有类型的bug吗？**
A: 适用于前端、后端、配置类的代码bug。不适用于硬件问题、第三方服务故障等非代码问题。

**Q: 修复代码会自动应用吗？**
A: 不会，会生成修复方案供您审核。您确认后再应用修改。

**Q: 如果修复引入新问题怎么办？**
A: fix-verification阶段会检查是否引入新问题，如果发现会重新生成修复方案。

**Q: 可以中途暂停吗？**
A: 可以，任务状态会保存在task_skill.md中，下次继续执行。

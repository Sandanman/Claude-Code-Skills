# Performance Analysis 原子Skill

## 概述
分析代码的性能瓶颈，包括渲染性能问题、执行效率问题和内存问题，基于代码质量分析结果进一步深入定位性能热点。为代码优化提供性能维度的数据支撑。

## 核心能力
- 分析渲染性能问题（重渲染、虚拟DOM更新）
- 检测执行效率问题（重复计算、大循环、同步阻塞）
- 识别内存问题（内存泄漏、闭包引用、大对象）
- 评估算法时间复杂度
- 提供具体的性能优化建议

## 分析维度

### 1. 渲染性能
- **组件重渲染**：响应式数据变化触发的组件重渲染次数
  - 未使用computed导致每次都重新计算
  - 对象字面量每次创建导致子组件重渲染
  - 未使用shallowRef/markRaw的大对象
- **虚拟DOM性能**
  - 模板中复杂的表达式
  - 频繁变化的大列表（未虚拟化）
  - 条件渲染导致的DOM频繁创建销毁
- **首屏加载性能**
  - 大组件同步加载
  - 未使用异步组件
  - 过多的同步import

### 2. 执行效率
- **重复计算**
  - 渲染函数中调用复杂计算（每次渲染都执行）
  - 循环内的重复计算可以提取到循环外
  - 相同的API调用重复发起
- **低效算法**
  - O(n²)复杂度以上的算法
  - 嵌套循环遍历同一数据集
  - 使用低效的数据结构查找（如数组indexOf代替Map）
- **同步阻塞**
  - 大循环在主线程执行阻塞UI
  - 未使用Web Worker处理大数据
  - 同步操作阻塞事件循环

### 3. 内存问题
- **内存泄漏**
  - 未清理的事件监听器（addEventListener无对应removeEventListener）
  - 未清理的定时器（setInterval/setTimeout无对应clear）
  - 全局变量引用组件实例导致无法回收
  - 闭包持有大对象引用
- **大对象**
  - 大数组/大对象未分页加载
  - 图片/媒体文件未压缩或懒加载
  - 缓存数据无上限增长

### 4. 计算复杂度评估
| 操作类型 | 复杂度 | 优化方向 |
|----------|--------|----------|
| 单次查找/访问 | O(1) | 理想状态 |
| 遍历数组查找 | O(n) | 使用Map缓存 |
| 嵌套循环遍历 | O(n²) | 优化算法结构 |
| 递归树遍历 | O(2^n) | 使用记忆化/迭代 |

- code-quality-analysis输出的质量分析报告
- 目标代码文件（用于深入分析）

分析src/components/MeetingCard.vue的性能瓶颈

## 瓶颈列表

### 瓶颈1：重复日期格式化（优先级：高）

**类型**：执行效率 - 重复计算
**影响指标**：渲染时间

#### 位置
- src/components/MeetingCard.vue:45-48
- src/components/MeetingCard.vue:89-92

#### 问题描述
在模板中使用函数调用formatDate()，每次渲染都会执行格式化逻辑，即使日期未变化。

```javascript
// 问题代码
const formattedStartTime = formatDate(meeting.startTime) // 每次渲染都执行
```

#### 性能影响
- 每次渲染额外执行2次日期格式化
- 每次格式化创建新的Date对象
- 估计增加渲染时间：2-3ms/次

#### 优化建议
使用computed属性缓存计算结果：
```javascript
const formattedStartTime = computed(() => formatDate(props.meeting.startTime))
```

#### 预期收益
- 减少重复计算：渲染次数从2n降至n（缓存命中时为0）
- 减少Date对象创建：每次渲染减少2个
- 性能提升：约5-10%

---

### 瓶颈2：响应式大对象未优化（优先级：高）

**类型**：渲染性能 - 重渲染

#### 位置
- src/components/MeetingCard.vue:30
- 父组件传递meeting对象

#### 问题描述
meeting对象包含大量属性，但只有部分用于当前组件展示。每次父组件任何属性变化都会导致当前组件重渲染。

```javascript
// 问题代码
const props = defineProps({
  meeting: Object  // 整个对象作为响应式，任何属性变化都触发重渲染
})
```

#### 性能影响
- 父组件meeting对象任意属性变化都触发重渲染
- 估计重渲染次数增加：3-5次/分钟
- 每次重渲染约15-20ms

#### 优化建议
使用toRefs + 解构，或使用shallowRef：
```javascript
const { title, status, startTime } = toRefs(props.meeting)
// 或
const props = defineProps({
  title: String,
  status: String,
  startTime: String,
  // 只暴露需要的属性
})
```

#### 预期收益
- 减少不必要的重渲染
- 渲染性能提升10-15%

### 瓶颈3：大列表未虚拟化（优先级：中）

**类型**：渲染性能 - DOM节点过多

#### 位置
- src/components/MeetingList.vue:20-80（如果存在）

#### 问题描述
使用v-for渲染超过100个会议项时，每个会议项都创建完整的DOM节点，导致初始渲染和滚动性能差。

#### 性能影响
- 初始渲染DOM节点过多
- 滚动时重新渲染所有可见项
- 内存占用大

#### 优化建议
使用虚拟滚动（如vue-virtual-scroller或动态导入分页加载）

#### 预期收益
- DOM节点从N个减少至可视数量（约20-30个）
- 滚动性能提升显著

---

### 瓶颈4：未清理的事件监听器（优先级：中）

**类型**：内存问题 - 内存泄漏

#### 位置
- src/components/MeetingCard.vue:40-45

#### 问题描述
组件中注册了window.resize事件监听，但组件销毁时未清理。

```javascript
// 问题代码
onMounted(() => {
  window.addEventListener('resize', handleResize)
})
// 缺少 onUnmounted 清理
```

#### 内存影响
- 每次组件创建都添加新监听器
- 组件销毁后监听器仍然存在
- 长期使用后内存泄漏

#### 优化建议
在onUnmounted中清理：
```javascript
onUnmounted(() => {
  window.removeEventListener('resize', handleResize)
})
```

#### 预期收益
- 消除内存泄漏风险
- 长时间使用内存稳定

## 执行逻辑
1. 接收code-quality-analysis的质量分析报告
2. 阅读目标代码文件，深入分析性能热点
3. 识别渲染性能瓶颈（重渲染、虚拟DOM问题）
4. 识别执行效率瓶颈（重复计算、低效算法）
5. 识别内存问题（泄漏、大对象）
6. 评估每个瓶颈的优化收益
7. 按优先级排序
8. 输出详细的性能瓶颈报告

## 依赖关系
- 依赖：code-quality-analysis（需要质量分析报告作为基础）
- 被依赖：improvement-suggestion

## 完成标准
1. 至少识别出3个性能瓶颈
2. 每个瓶颈包含位置、问题描述、性能影响
3. 包含优化建议和预期收益
4. 提供优先级排序（高/中/低）
5. 输出完整格式化的报告（markdown）

## 错误处理
- **代码文件不存在**：报错并退出
- **性能影响难以量化**：标注为"需要运行时测量"
- **无法确定瓶颈类型**：标记为"需要profiling工具分析"

## 注意事项
- 性能分析基于静态代码分析，具体性能影响需要运行时验证
- 某些"瓶颈"在数据量小时不明显，数据量大时才暴露
- 优化时需权衡代码可读性和性能
- 优先处理高优先级瓶颈（影响每次渲染/调用的）

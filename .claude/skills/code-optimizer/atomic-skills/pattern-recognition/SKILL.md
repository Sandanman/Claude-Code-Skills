# Pattern Recognition 原子Skill

## 概述
基于代码质量分析报告，识别具体的可优化模式，定位代码中的反模式和低效实现，为后续生成优化建议提供具体目标。

## 核心能力
- 识别低效算法模式（时间复杂度优化）
- 检测冗余逻辑模式（重复计算、冗余判断）
- 发现不良实践模式（反模式、违反原则）
- 定位性能瓶颈（频繁DOM操作、过度渲染、内存泄漏）
- 提取可复用模式（组件拆分、函数提取）

## 支持的模式类型

### 1. 性能相关模式
- **低效算法**
  - O(n²) → O(n)：嵌套循环可优化为单层循环或使用Map
  - 重复计算：相同计算在循环内重复执行
  - 不必要的遍历：可以使用find/findOne替代forEach

- **内存泄漏**
  - 未清理的事件监听器
  - 未清理的定时器
  - 闭包持有大对象引用

- **渲染性能**
  - 频繁的DOM操作
  - 大列表未虚拟化
  - 未使用memo/useMemo缓存
  - 不必要的组件重渲染

### 2. 可维护性相关模式
- **大组件模式**
  - 单个组件超过300行
  - 组件职责过多（渲染+逻辑+数据获取）
  - 可拆分为多个子组件

- **重复逻辑模式**
  - 相同的条件判断在多处出现
  - 相同的数据转换逻辑
  - 相同的错误处理逻辑

- **过长函数模式**
  - 函数超过50行
  - 函数包含多个抽象层级
  - 函数有多个返回点

### 3. 代码质量相关模式
- **嵌套条件模式**
  - 三重以上if-else嵌套
  - 可使用卫语句简化
  - 可使用早返回减少嵌套

- **参数过多模式**
  - 函数参数超过5个
  - 可使用对象封装参数
  - 可使用配置对象模式

- **硬编码模式**
  - 魔术数字未定义常量
  - 魔术字符串未定义枚举
  - 配置项硬编码在代码中

### 4. 安全相关模式
- **XSS风险模式**
  - 使用v-html或innerHTML渲染用户输入
  - 未对用户输入进行sanitization

- **不安全API模式**
  - 使用eval()
  - 使用document.write()
  - 直接拼接SQL语句

### 模式1：低效算法 - 重复日期格式化（优先级：🔴 高）

**模式类型**：性能相关
**严重程度**：高
**影响范围**：性能

#### 位置
- 第45-48行：`formatDate(meeting.startTime)`
- 第89-92行：`formatDate(meeting.endTime)`

#### 问题描述
在渲染函数中，每次渲染都会重新格式化日期，即使日期未改变。该格式化函数在组件中有2处重复调用。

```javascript
// 问题代码
const formattedStartTime = formatDate(meeting.startTime) // 第45行
const formattedEndTime = formatDate(meeting.endTime) // 第89行

function formatDate(timestamp) {
  const date = new Date(timestamp)
  return `${date.getFullYear()}-${date.getMonth()+1}-${date.getDate()} ${date.getHours()}:${date.getMinutes()}`
}
```

#### 根本原因
- 日期格式化是纯函数，相同输入总是产生相同输出
- 未使用缓存机制，每次调用都重新计算
- 逻辑重复，在多处调用相同的格式化

#### 模式分类
- **反模式**：重复计算
- **优化方向**：提取为computed属性或使用缓存

#### 预期收益
- 性能提升：减少不必要的日期对象创建，提升5-10%
- 代码复用：提取为共享函数，减少重复代码

#### 检测依据
- 静态分析检测到formatDate函数在多处被调用
- 相同的参数传递模式（meeting.startTime, meeting.endTime）
- 重复代码检测显示相似度92%

### 模式2：大函数模式 - handleJoinMeeting过长（优先级：🔴 高）

**模式类型**：可维护性相关
**严重程度**：高
**影响范围**：可维护性、可测试性

#### 位置
- 第78-125行：`handleJoinMeeting`方法

#### 问题描述
单个方法48行，包含多个职责：
- 参数验证（5行）
- 权限检查（8行）
- 网络请求（10行）
- 错误处理（12行）
- 状态更新（8行）
- 页面跳转（5行）

```javascript
// 问题代码片段
const handleJoinMeeting = async (meetingId, password, isAudioOnly, isVideoOn, userId, userName) => {
  // 参数验证（5行）
  if (!meetingId || !userId) { ... }

  // 权限检查（8行）
  const hasPermission = await checkPermission(userId, meetingId)
  if (!hasPermission) { ... }

  // 网络请求（10行）
  try {
    const response = await joinMeetingAPI({ ... })
    ...
  } catch (error) {
    // 错误处理（12行）
    ...
  }

  // 状态更新（8行）
  meetingStore.updateStatus(...)

  // 页面跳转（5行）
  router.push(...)
}
```

#### 根本原因
- 违反单一职责原则（SRP）
- 函数包含多个抽象层级
- 业务逻辑、数据获取、UI更新耦合在一起

#### 模式分类
- **反模式**：大函数、职责混乱
- **优化方向**：拆分为多个小函数，分离关注点

#### 预期收益
- 可维护性：每个函数职责清晰，更容易理解和修改
- 可测试性：小函数更容易单独测试
- 可复用性：拆分出的函数可在其他地方复用

#### 检测依据
- 圈复杂度=22（远超目标值15）
- 函数行数=48（接近50行边界）
- 包含多个try-catch和条件判断

### 模式3：嵌套过深 - 多层条件判断（优先级：🟡 中）

**模式类型**：代码质量相关
**严重程度**：中
**影响范围**：可读性、可维护性

#### 位置
- 第95-110行：会议状态判断逻辑

#### 问题描述
三重if-else嵌套，逻辑复杂，难以理解：

```javascript
// 问题代码
if (meeting.status === 'ongoing') {
  if (meeting.hasPassword) {
    if (meeting.isLocked) {
      // 锁定状态
    } else {
      // 未锁定但需要密码
    }
  } else {
    // 无需密码
  }
} else if (meeting.status === 'pending') {
  // ...
}
```

#### 根本原因
- 使用嵌套if-else处理多条件判断
- 未使用早返回或卫语句
- 条件判断层级过多

#### 模式分类
- **反模式**：箭头代码（嵌套过深）
- **优化方向**：使用卫语句、策略模式或状态机

#### 预期收益
- 可读性：代码更扁平，逻辑更清晰
- 维护性：每个条件独立，更容易修改

#### 检测依据
- 嵌套深度=3（超过目标值2）
- 圈复杂度贡献=8

**模式类型**：代码质量相关

### 模式5：XSS风险 - v-html渲染用户输入（优先级：🔴 高）

**模式类型**：安全相关
**严重程度**：高
**影响范围**：安全性

#### 位置
- 第145行：`<div v-html="meeting.description"></div>`

#### 问题描述
直接使用v-html渲染用户输入的会议描述，可能导致XSS攻击：

```javascript
// 问题代码
<div v-html="meeting.description"></div>
```

如果`meeting.description`包含恶意脚本：
```html
<script>alert('XSS')</script>
```

#### 根本原因
- 未对用户输入进行sanitization
- Vue的v-html会直接插入HTML，不会转义

#### 模式分类
- **安全漏洞**：XSS注入风险
- **优化方向**：使用{{ }}插值或使用sanitize-html库

#### 预期收益
- 安全性：防止XSS攻击
- 合规性：符合安全编码标准

#### 检测依据
- 使用v-html指令
- 绑定的是用户输入字段
- 未使用sanitization函数
